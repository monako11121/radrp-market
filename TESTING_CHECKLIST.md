# RADRP Market — Чеклист тестирования

> Для закрытого тестирования. Тестировать в браузере на http://localhost:3000  
> Перед тестом: убедись, что у тестовых пользователей есть баланс (через Prisma Studio).

---

## Подготовка

- [ ] Запустить `npm run dev`
- [ ] Создать минимум **2 тестовых аккаунта** (UserA и UserB)
- [ ] Выдать обоим баланс через Prisma Studio (`npx prisma studio` → таблица User)
- [ ] Назначить себя администратором — добавить email в `ADMIN_EMAILS` в `.env`

---

## 1. Регистрация

| Шаг | Ожидаемый результат | ✓/✗ |
|---|---|---|
| Перейти на `/auth`, вкладка "Регистрация" | Форма с полями username, email, пароль | |
| Оставить поля пустыми → "Зарегистрироваться" | Ошибка "Заполните все поля" | |
| Ввести пароль короче 8 символов | Ошибка "Пароль должен содержать минимум 8 символов" | |
| Зарегистрировать UserA (валидные данные) | Редирект на страницу входа | |
| Зарегистрировать UserA повторно с тем же email | Ошибка "Пользователь с таким email уже зарегистрирован" | |
| Попытаться занять username UserA другим email | Ошибка "Это имя пользователя уже занято" | |
| Зарегистрировать UserB (другой email/username) | Успех, редирект на вход | |

---

## 2. Вход

| Шаг | Ожидаемый результат | ✓/✗ |
|---|---|---|
| Войти как UserA (корректные данные) | Редирект на главную, имя/баланс в Header | |
| Войти с неверным паролем | Остаться на странице входа (NextAuth не перенаправляет) | |
| Войти с несуществующим email | Остаться на странице входа | |
| Проверить Header | Отображается актуальный баланс из БД, первая буква имени | |
| Нажать "Выйти" | Редирект на главную, Header показывает "Войти" | |

---

## 3. Создание объявления

> Войти как UserA

| Шаг | Ожидаемый результат | ✓/✗ |
|---|---|---|
| Нажать "Продать" в Header (не авторизован) | Редирект на `/auth` | |
| Войти и нажать "Продать" | Открывается `/sell` с формой | |
| Отправить форму с пустым названием | Ошибка "Введите название товара" | |
| Указать цену = 0 | Ошибка "Цена должна быть больше 0" | |
| Указать сервер 22 (или 0) | Ошибка "Сервер должен быть от 1 до 21" | |
| Выбрать категорию "Вирты" | Появляются доп. поля: количество кк и цена за 1кк | |
| Заполнить все поля корректно → "Опубликовать" | Редирект на `/catalog`, товар виден в каталоге | |
| Открыть карточку своего товара | Кнопка "Купить" отсутствует, показывается "Это ваш товар" | |

---

## 4. Покупка (создание сделки)

> UserA — продавец, UserB — покупатель

| Шаг | Ожидаемый результат | ✓/✗ |
|---|---|---|
| Войти как UserB | — | |
| Найти товар UserA в каталоге, открыть карточку | Кнопка "Купить" видна | |
| Нажать "Купить" | Редирект на `/deals`, новая сделка в статусе "Ожидание" | |
| Нажать "Купить" ещё раз на том же товаре | Редирект на **уже существующую** сделку (не создаётся дубликат) | |
| Войти как UserA → `/deals` | Сделка с UserB отображается | |

---

## 5. Чат в сделке

> В сделке между UserA и UserB

| Шаг | Ожидаемый результат | ✓/✗ |
|---|---|---|
| UserB: написать сообщение | Сообщение появляется (справа, оранжевый фон) | |
| UserA: открыть ту же сделку | Сообщение UserB видно (слева, тёмный фон) | |
| UserA: ответить | Ответ виден обоим | |
| Отправить пустое сообщение | Ничего не происходит | |
| Попытаться написать в чужую сделку напрямую через API | Сообщение не создаётся (проверка участника) | |

---

## 6. Завершение сделки

| Шаг | Ожидаемый результат | ✓/✗ |
|---|---|---|
| UserA: принять сделку ("Принять") | Статус меняется на "В работе" | |
| UserA: в статусе IN_PROGRESS кнопок нет кроме "Спор" | Кнопка "Завершить" у UserA отсутствует | |
| UserB: нажать "Завершить" | — | |
| Если у UserB баланс < цены товара | Ошибка "Недостаточно средств" | |
| Если баланс достаточный | Статус → "Завершено", баланс UserB уменьшился, UserA — вырос | |
| Обновить Header (перейти на другую страницу) | Баланс в Header актуальный | |
| Нажать "Завершить" ещё раз | Ничего не происходит (идемпотентность) | |
| Кнопки на завершённой сделке | Кнопок нет совсем | |

---

## 7. Спор

| Шаг | Ожидаемый результат | ✓/✗ |
|---|---|---|
| В активной сделке нажать "Спор" | Статус → "Спор", в сделке показывается "Спор открыт" | |
| Открыть `/disputes` как UserA | Только сделки UserA (buyer или seller) | |
| Открыть `/disputes` как UserB | Только сделки UserB | |
| Войти как третий пользователь (UserC) | `/disputes` пустой (чужие споры не видны) | |
| UserA: "Завершить и оплатить" | Баланс списывается, статус → Завершено | |
| UserA: "Вернуть в работу" | Статус → В работе | |
| Попытаться действовать по чужому спору через DevTools | Action возвращает "Нет доступа" | |

---

## 8. Админка споров

> Войти под email, указанным в `ADMIN_EMAILS`

| Шаг | Ожидаемый результат | ✓/✗ |
|---|---|---|
| Открыть `/admin/disputes` | Список всех DISPUTE-сделок из БД | |
| Войти под не-admin аккаунтом → `/admin/disputes` | Редирект на `/` | |
| Нажать "Покупатель прав" | Сделка → DONE, деньги не двигаются | |
| Нажать "Продавец прав" при нехватке средств | Ошибка "Недостаточно средств" | |
| Нажать "Продавец прав" при достаточном балансе | Деньги переведены, сделка закрыта | |
| Дважды нажать кнопку за раз (race condition) | Только одна операция проходит | |

---

## 9. Профиль

| Шаг | Ожидаемый результат | ✓/✗ |
|---|---|---|
| Открыть `/profile` | Реальное имя и баланс из БД | |
| Список объявлений | Только свои товары | |
| Нажать "Удалить" на товаре без активных сделок | Товар удаляется | |
| Нажать "Удалить" на товаре с активной сделкой | Ошибка "Нельзя удалить товар с активной сделкой" | |
| Нажать "Редактировать" | Форма с текущими данными товара | |
| Изменить цену и сохранить | Изменения видны в каталоге | |

---

## 10. Настройки профиля

| Шаг | Ожидаемый результат | ✓/✗ |
|---|---|---|
| Открыть `/profile/settings` | Текущий username и email (email не редактируется) | |
| Оставить поле username пустым | Ошибка "Username не может быть пустым" | |
| Ввести username короче 3 символов | Ошибка "минимум 3 символа" | |
| Ввести занятый username | Ошибка "Это имя пользователя уже занято" | |
| Сохранить тот же username | Сообщение "Username не изменился" | |
| Ввести новый уникальный username → "Сохранить" | Зелёное "Username успешно обновлён" | |

---

## Известные ограничения (не баги)

- Баланс в Header обновляется при навигации, не в реальном времени
- После смены username имя в аватаре Header обновится только при повторном входе
- Пополнение баланса — только через Prisma Studio (UI не реализован)
- "Причина спора" — статичный текст, не хранится в БД
- Публичные профили продавцов не реализованы
